## 1. Business Context & Problem Statement

### Who is Nomos?
Nomos is a B2B white-label energy infrastructure company. They enable non-energy companies — EV manufacturers, heat pump installers, battery storage OEMs — to offer their own branded dynamic electricity tariffs to end customers. Nomos handles procurement, billing, and customer support behind the scenes. A partner (e.g. "Müller Wärmepumpen GmbH") plugs into Nomos via API and within 7 days can offer its customers a full energy plan under their own brand.

### The Problem NomosFlow Solves
When an energy retailer operates at scale, three things break their unit economics:

1. **Failed payments / dunning failures** — A customer's direct debit fails. The retailer has no intelligent system to decide: when to retry, what message to send, whether this customer is about to churn or just had a bad month. Most energy retailers handle this with manual CSR queues and generic email blasts. Industry average: 3–8% of monthly revenue in energy retail is at risk from failed payment cycles.

2. **Reactive churn management** — By the time a customer cancels, it's too late. Most energy retailers have no predictive signal. They only discover churn after it's happened. The customer was recoverable at day 7; they're gone by day 30.

3. **Generic outreach** — A boilerplate "your payment failed" email has ~12% open rates and drives cancellations. A personalised message tied to the customer's device (EV, heat pump), their usage savings, and their upcoming cheap tariff windows has demonstrated 3× better retention outcomes in comparable fintech/utilities contexts.

NomosFlow is the operational back-office intelligence layer that solves all three — powered by Gemini AI. It is the kind of zero-to-one internal tooling Nomos needs to build as they scale their partner network.

***

## 2. Application Overview

**NomosFlow** is a full-stack web application simulating the internal operations dashboard that a Nomos partner success team or OEM partner would use to manage their energy customer base. It demonstrates:

- Intelligent dunning: classify failures, schedule smart retries, track resolution
- AI churn scoring: predict which customers are at risk before they leave
- AI retention messaging: personalised, device-aware outreach generated by Gemini
- White-label partner view: switch between partner brands to demonstrate Nomos's OEM proposition
- Real-time AI pipeline: trigger the full dunning cycle live and watch Gemini process customers in sequence

- **Data Cards Row**: Top of each page has a row of 4 summary stat cards — bold large number, small muted label beneath, subtle icon.
- **Slide-out Drawer**: Customer detail view opens as a right-side drawer (not a new page) — `480px` wide, overlays main content with backdrop.

## 5. Database Schema (PostgreSQL)

```sql
-- ================================================================
-- PARTNERS: OEM white-label clients of Nomos
-- ================================================================
CREATE TABLE partners (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) NOT NULL,           -- "Müller Wärmepumpen GmbH"
  slug            VARCHAR(100) UNIQUE NOT NULL,    -- "muller-wp" (used in UI switcher)
  device_type     VARCHAR(50)  NOT NULL,           -- 'ev' | 'heat_pump' | 'battery' | 'mixed'
  brand_color     VARCHAR(7)   DEFAULT '#3D6B2C',  -- hex for white-label accent theming
  logo_url        TEXT,
  customer_count  INT GENERATED ALWAYS AS STORED, -- virtual, computed from customers table
  created_at      TIMESTAMP    DEFAULT NOW()
);

-- ================================================================
-- CUSTOMERS: End customers belonging to a partner
-- ================================================================
CREATE TABLE customers (
  id                UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  partner_id        UUID      NOT NULL REFERENCES partners(id) ON DELETE CASCADE,
  name              VARCHAR(255) NOT NULL,
  email             VARCHAR(255) NOT NULL,
  phone             VARCHAR(50),
  tariff_type       VARCHAR(50)  NOT NULL DEFAULT 'dynamic', -- 'dynamic' | 'fixed'
  device_type       VARCHAR(50)  NOT NULL,                   -- 'ev' | 'heat_pump' | 'battery'
  monthly_kwh       DECIMAL(10,2),                           -- avg monthly consumption
  annual_saving_eur DECIMAL(10,2),                           -- projected vs fixed tariff
  salary_day        INT          CHECK (salary_day BETWEEN 1 AND 31), -- for smart retry
  contract_start    DATE         NOT NULL,
  contract_status   VARCHAR(50)  NOT NULL DEFAULT 'active',  -- 'active' | 'suspended' | 'cancelled'
  city              VARCHAR(100),
  created_at        TIMESTAMP    DEFAULT NOW()
);

-- ================================================================
-- PAYMENTS: Monthly billing records
-- ================================================================
CREATE TABLE payments (
  id               UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id      UUID      NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
  amount_eur       DECIMAL(10,2) NOT NULL,
  period_month     DATE      NOT NULL,       -- first day of billing month
  due_date         DATE      NOT NULL,
  paid_at          TIMESTAMP,
  status           VARCHAR(50) NOT NULL DEFAULT 'pending',
    -- 'pending' | 'paid' | 'failed' | 'retrying' | 'written_off'
  failure_reason   VARCHAR(100),
    -- 'insufficient_funds' | 'expired_card' | 'bank_block' | 'sepa_reject' | 'unknown'
  failure_classified_by VARCHAR(20) DEFAULT 'manual',
    -- 'manual' | 'ai' (set when Gemini classifies it)
  retry_count      INT          DEFAULT 0,
  max_retries      INT          DEFAULT 3,
  next_retry_date  DATE,
  created_at       TIMESTAMP    DEFAULT NOW()
);

-- ================================================================
-- DUNNING ACTIONS: Every step taken in the dunning process
-- ================================================================
CREATE TABLE dunning_actions (
  id                    UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  payment_id            UUID      NOT NULL REFERENCES payments(id) ON DELETE CASCADE,
  customer_id           UUID      NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
  action_type           VARCHAR(50) NOT NULL,
    -- 'ai_classify' | 'retry_scheduled' | 'retry_attempted' | 'email_sent'
    -- | 'sms_sent' | 'suspension_warning' | 'suspended' | 'resolved' | 'written_off'
  ai_generated_message  TEXT,                   -- populated for email/sms types
  ai_failure_reason     VARCHAR(100),           -- populated for ai_classify type
  ai_confidence         DECIMAL(4,3),           -- 0.000 - 1.000
  retry_scheduled_for   DATE,                   -- populated for retry_scheduled type
  triggered_by          VARCHAR(50) DEFAULT 'system', -- 'system' | 'manual'
  outcome               VARCHAR(50),
    -- 'success' | 'failed_again' | 'no_response' | 'cancelled_by_customer' | 'pending'
  executed_at           TIMESTAMP  DEFAULT NOW()
);

-- ================================================================
-- CHURN SCORES: AI-generated churn risk per customer (latest wins)
-- ================================================================
CREATE TABLE churn_scores (
  id              UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id     UUID      NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
  score           INT       NOT NULL CHECK (score BETWEEN 0 AND 100),
  risk_level      VARCHAR(20) NOT NULL,
    -- 'low' (0-25) | 'medium' (26-50) | 'high' (51-75) | 'critical' (76-100)
  reasoning       TEXT,                        -- Gemini's explanation in plain English
  factors         JSONB,
    -- {
    --   "failed_payments_30d": 2,
    --   "days_since_last_paid": 18,
    --   "retry_exhaustion_pct": 0.66,
    --   "contract_age_days": 45,
    --   "device_type": "heat_pump",
    --   "is_dynamic_tariff": true
    -- }
  action_suggested TEXT,                       -- Gemini's recommended next action
  scored_at       TIMESTAMP  DEFAULT NOW()
);

-- ================================================================
-- INDEXES
-- ================================================================
CREATE INDEX idx_customers_partner     ON customers(partner_id);
CREATE INDEX idx_customers_status      ON customers(contract_status);
CREATE INDEX idx_payments_customer     ON payments(customer_id);
CREATE INDEX idx_payments_status       ON payments(status);
CREATE INDEX idx_dunning_customer      ON dunning_actions(customer_id);
CREATE INDEX idx_dunning_payment       ON dunning_actions(payment_id);
CREATE INDEX idx_churn_customer        ON churn_scores(customer_id);
CREATE INDEX idx_churn_scored_at       ON churn_scores(scored_at DESC);
```

import google.generativeai as genai
from app.config import settings

genai.configure(api_key=settings.gemini_api_key)
model = genai.GenerativeModel("gemini-1.5-flash")


def classify_payment_failure(customer: dict, payment: dict) -> dict:
    """
    Classifies the most likely reason a payment failed using customer context.
    Returns: { failure_reason: str, confidence: float, explanation: str }
    """
    prompt = f"""
You are an expert operations analyst at a European energy retailer.

A customer's direct debit payment has failed. Based on the context below, 
classify the most likely reason for the failure.

CUSTOMER CONTEXT:
- Name: {customer['name']}
- Device type: {customer['device_type']}  (ev / heat_pump / battery)
- Contract started: {customer['contract_start']}
- Contract age (days): {customer['contract_age_days']}
- Payment history: {customer['payment_history_summary']}
  (e.g. "3 of last 4 payments succeeded, 1 failed")
- Usual salary day: {customer.get('salary_day', 'unknown')}
- Current day of month payment failed: {payment['day_of_month_failed']}
- Amount (EUR): {payment['amount_eur']}
- Retry count so far: {payment['retry_count']}

POSSIBLE FAILURE REASONS:
1. insufficient_funds — account balance too low, common near end of month
2. expired_card — card on file has expired (more likely for older contracts)  
3. bank_block — bank rejected the debit mandate (common for new contracts <60 days)
4. sepa_reject — SEPA mandate was revoked by the customer
5. unknown — not enough signal to classify

Respond ONLY as valid JSON:
{{
  "failure_reason": "<one of the 5 options above>",
  "confidence": <float 0.0-1.0>,
  "explanation": "<one sentence plain English explanation>"
}}
"""
    response = model.generate_content(prompt)
    import json
    return json.loads(response.text.strip().strip("```json").strip("```"))


def score_churn_risk(customer: dict, payment_history: list,
